## Overview

**PicoClaw** is an ultra-lightweight personal AI assistant written entirely in Go from scratch. It was inspired by NanoBot and designed to run on minimal hardware ($10 devices with <10MB RAM) while providing powerful AI agent capabilities.

### Key Characteristics

- **Ultra-lightweight**: Core memory footprint <10MB
- **Fast boot**: Starts in <1s even on 0.6GHz single-core processors
- **Multi-architecture**: Runs on x86, ARM, MIPS, RISC-V, LoongArch
- **AI-bootstrapped**: 95% of core code was generated by AI agents
- **Multi-platform**: Supports 15+ chat platforms (Telegram, Discord, Slack, WhatsApp, etc.)
- **Multi-provider**: Supports 10+ LLM providers (OpenAI, Anthropic, AWS Bedrock, etc.)
- **Tool system**: Extensible tool registry for file operations, shell commands, web search, etc.
- **MCP support**: Native Model Context Protocol integration

## Architecture Overview

PicoClaw follows a modular, event-driven architecture with clear separation of concerns:

```
┌─────────────────────────────────────────────────────────────────┐
│                         Gateway (Entry Point)                    │
│  - Loads configuration                                           │
│  - Initializes all services                                      │
│  - Manages lifecycle                                             │
└───────────────┬────────────────────────────────┬─────────────────┘
                │                                │
        ┌───────▼─────────┐          ┌───────▼──────────┐
        │  Channel Manager │          │   Agent Loop     │
        │  - Telegram      │          │  - Process msgs  │
        │  - Discord       │◄────────►│  - Tool calls    │
        │  - Slack, etc.   │          │  - LLM interact  │
        └────────┬─────────┘          └──────┬───────────┘
                 │                           │
                 │      ┌────────────────────┤
                 │      │                    │
          ┌──────▼──────▼──────┐     ┌──────▼──────────┐
          │    Message Bus      │     │   Provider      │
          │  - Inbound queue    │     │  - OpenAI       │
          │  - Outbound queue   │     │  - Anthropic    │
          │  - Media queue      │     │  - Bedrock, etc.│
          └─────────┬───────────┘     └─────────────────┘
                    │
          ┌─────────▼──────────────────────┐
          │        Services                │
          │  - Memory Store (Sessions)     │
          │  - Media Store                 │
          │  - Tool Registry               │
          │  - MCP Runtime                 │
          │  - Cron Scheduler              │
          │  - State Manager               │
          └────────────────────────────────┘
```

### Request Flow

1. **Inbound**: User sends message → Channel adapter receives → Publishes to Message Bus
2. **Processing**: Agent Loop consumes → Loads session history → Calls LLM with tools
3. **Tool Execution**: LLM requests tools → Agent executes → Results returned to LLM
4. **Outbound**: LLM generates response → Published to Message Bus → Channel adapter sends to user

## Core Components

### 1. Agent System

**Location**: `pkg/agent/`

The Agent is the core intelligence layer that orchestrates the conversation flow, tool execution, and LLM interactions.

#### Key Files

- `instance.go`: Defines `AgentInstance` with workspace, tools, provider, sessions
- `loop.go`: Main agent loop that processes messages (`AgentLoop`)
- `context.go`: Builds conversation context with history, system prompts, memory
- `turn.go`: Represents a single conversation turn
- `subturn.go`: Nested agent invocations (sub-agents)
- `hooks.go`: Extensible hook system for custom logic

#### AgentInstance Structure

```go
type AgentInstance struct {
    ID                string
    Name              string
    Model             string
    Workspace         string
    MaxIterations     int           // Max tool call iterations
    MaxTokens         int           // Context window size
    Temperature       float64
    Provider          providers.LLMProvider
    Sessions          session.SessionStore
    ContextBuilder    *ContextBuilder
    Tools             *tools.ToolRegistry
    Router            *routing.Router  // Model routing
    Subagents         *config.SubagentsConfig
}
```

#### Agent Loop Flow

```
User Message
     ↓
Load Session History
     ↓
Build Context (system prompt + history + user message)
     ↓
Call LLM with Tools
     ↓
┌────────────────────────────┐
│  LLM Returns Response      │
│  - Text only?  → Send      │
│  - Tool calls? → Execute   │
└────────┬───────────────────┘
         │
    Tool Execution
         ↓
   Call LLM Again (iteration)
         ↓
   [Repeat until text response or max iterations]
```

#### Context Builder

The `ContextBuilder` assembles the conversation context:

- System prompt from `AGENT.md`, `SOUL.md`, `USER.md`, `IDENTITY.md`
- Conversation history from memory store
- Tool definitions
- Budget tracking (token limits)
- Dynamic context (sender info, skills)

#### Event System & Hooks

PicoClaw has a powerful hook system for extending behavior:

```go
type EventBus interface {
    Emit(event Event)
    Subscribe(eventType EventType, handler EventHandler)
}

// Hook types: before_turn, after_turn, before_tool, after_tool, etc.
```

### 2. Provider System (LLM Integration)

**Location**: `pkg/providers/`

#### Provider Interface

```go
type LLMProvider interface {
```

#### Supported Providers

- **OpenAI**: GPT-4, GPT-3.5, etc. (`openai_compat/`)
- **Anthropic**: Claude models (`anthropic/`, `anthropic_messages/`)
- **AWS Bedrock**: Claude via AWS (`bedrock/`)
- **Azure OpenAI**: Azure-hosted models (`azure/`)
- **GitHub Copilot**: `github_copilot_provider.go`
- **Antigravity**: Custom provider (`antigravity_provider.go`)
- **CLI Providers**: Local Claude/Codex executables

#### Fallback Chain

PicoClaw supports automatic failover between models:

```go
type FallbackChain struct {

// Tracks cooldowns to avoid repeatedly hitting failed models
```

Providers classify errors for intelligent failover:

```go
type FailoverReason string
```

### 4. Tools System

**Location**: `pkg/tools/`

Tools are actions the LLM can invoke to interact with the environment.

#### Tool Interface

```go
type Tool interface {
    Name() string
    Description() string
    Parameters() map[string]any  // JSON Schema
    Execute(ctx context.Context, args map[string]any) (ToolResult, error)
}
```

#### Built-in Tools

| Tool              | Description                         |
| ----------------- | ----------------------------------- |
| `read_file`       | Read file contents                  |
| `write_file`      | Write/overwrite file                |
| `edit_file`       | String replacement in file          |
| `append_file`     | Append to file                      |
| `list_dir`        | List directory contents             |
| `exec` / `shell`  | Execute shell commands              |
| `web_search`      | Search the web (multiple providers) |
| `fetch_webpage`   | Download webpage content            |
| `send_message`    | Send message to another channel     |
| `spawn`           | Launch sub-agent                    |
| `spawn_status`    | Check sub-agent status              |
| `cron`            | Schedule recurring tasks            |
| `skill_search`    | Find available skills               |
| `skill_install`   | Install skill from URL              |
| `i2c_*` / `spi_*` | Hardware I/O (Linux)                |

#### Tool Registry

Tools are registered with the agent:

```go
type ToolRegistry struct {
```

#### Tool Result Types


#### Tool Security

- **Workspace restriction**: Tools can be restricted to agent workspace

### 5. Message Bus

**Location**: `pkg/bus/`

The message bus decouples channels from agents using an asynchronous pub-sub pattern.

#### Message Types

```go
type InboundMessage struct {
    Channel           string
    ChatID            string
    SenderID          string
    Content           string
    MessageID         string
    Media             []string  // media:// URIs
    Metadata          map[string]string
}

type OutboundMessage struct {
    Channel           string
    ChatID            string
    Content           string
    ReplyToMessageID  string
    EditMessageID     string  // Edit instead of send
    Metadata          map[string]string
}

type OutboundMediaMessage struct {
    Channel   string
    ChatID    string
    MediaRefs []string  // media:// URIs
    Caption   string
}
```

#### MessageBus API

```go
type MessageBus struct {
    inbound       chan InboundMessage
    outbound      chan OutboundMessage
    outboundMedia chan OutboundMediaMessage
}

func (mb *MessageBus) PublishInbound(ctx, msg) error
func (mb *MessageBus) PublishOutbound(ctx, msg) error
func (mb *MessageBus) InboundChan() <-chan InboundMessage
func (mb *MessageBus) OutboundChan() <-chan OutboundMessage
```

#### Flow

```
Channel Adapter → InboundMessage → Agent Loop
Agent Loop → OutboundMessage → Channel Adapter
```

### 6. Memory & Session Management

**Location**: `pkg/memory/`, `pkg/session/`

Manages conversation history and long-term memory.

#### Memory Store Interface

```go
type Store interface {
    AddMessage(ctx, sessionKey, role, content string) error
    AddFullMessage(ctx, sessionKey string, msg Message) error
    GetHistory(ctx, sessionKey string) ([]Message, error)
    GetSummary(ctx, sessionKey string) (string, error)
    SetSummary(ctx, sessionKey, summary string) error
    TruncateHistory(ctx, sessionKey string, keepLast int) error
    SetHistory(ctx, sessionKey string, history []Message) error
    Compact(ctx, sessionKey string) error
    Close() error
}
```

#### Backend: JSONL

Default implementation uses JSONL (JSON Lines) format:

```
~/.picoclaw/workspace/sessions/<session_key>.jsonl
```

Each line is a JSON object:

```json
{"role":"user","content":"Hello"}
{"role":"assistant","content":"Hi! How can I help?"}
{"role":"user","content":"Tell me about PicoClaw"}
```

#### Session Store

```go
type SessionStore interface {
    GetOrCreate(sessionKey string) (*Session, error)
    Get(sessionKey string) (*Session, error)
}
```

A `Session` wraps a `memory.Store` with additional logic:

- Message limits
- Token budgets
- Summarization triggers

#### Summarization

When history exceeds `SummarizeMessageThreshold`, PicoClaw:

1. Asks LLM to summarize old messages
2. Stores summary
3. Truncates old history
4. Keeps summary + recent messages

### 7. MCP Integration (Model Context Protocol)

**Location**: `pkg/mcp/`

PicoClaw natively supports the [Model Context Protocol](https://modelcontextprotocol.io/) for connecting to MCP servers.

#### MCP Manager

```go
type ServerConnection struct {
    Name    string
    Client  *mcp.Client
    Tools   []mcp.Tool
}

type Manager struct {
    servers     map[string]*ServerConnection
    toolMapping map[string]string  // tool name → server name
}

func (m *Manager) ConnectStdio(name, command string, args []string) error
func (m *Manager) ConnectSSE(name, url string, headers map[string]string) error
func (m *Manager) GetMCPTools() []ToolDefinition
func (m *Manager) ExecuteMCPTool(ctx, toolName, args) (result, error)
```

#### Configuration

```json
{
  "tools": {
    "mcp_servers": {
      "filesystem": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
        "disabled": false
      },
      "github": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "env": {
          "GITHUB_TOKEN": "${GITHUB_TOKEN}"
        }
      }
    }
  }
}
```

#### Integration Flow

1. Gateway starts MCP servers (stdio or SSE)
2. MCP Manager queries available tools
3. Tools registered in agent's ToolRegistry
4. LLM can call MCP tools like native tools
5. Agent routes tool calls to appropriate MCP server

### 8. Routing System

**Location**: `pkg/routing/`

Intelligently routes requests to different models based on complexity.

#### Router

```go
type Router struct {
    cfg        RouterConfig
    classifier Classifier
}

func (r *Router) SelectModel(
    msg string,
    history []Message,
    primaryModel string,
) (model string, usedLight bool, score float64)
```

#### Complexity Classification

Features extracted from message:

- **Length**: Word count
- **Code blocks**: Presence of ``` fenced code
- **URLs**: Link references
- **Media**: Image attachments
- **Technical terms**: Keywords like "debug", "implement"
- **History depth**: Recent conversation length

#### Example Configuration

```json
{
  "agents": {
    "defaults": {
      "model_name": "gpt-4o",
      "routing": {
        "light_model": "gpt-4o-mini",
        "threshold": 0.35
      }
    }
  }
}
```

**Logic**:

- Simple queries (score < 0.35) → `gpt-4o-mini`
- Complex queries (score ≥ 0.35) → `gpt-4o`

**Benefits**:

- Reduce API costs
- Faster responses for simple queries
- Heavy model only when needed

## Data Flow

### Complete Request Lifecycle

```
1. User sends message in Telegram
           ↓
2. Telegram channel adapter receives update
           ↓
3. Adapter publishes InboundMessage to bus
           ↓
4. Agent Loop consumes from inbound channel
           ↓
5. Load session history from memory store
           ↓
6. Build conversation context:
   - System prompt (AGENT.md, etc.)
   - History
   - User message
   - Tool definitions
           ↓
7. Call LLM Provider (e.g., OpenAI)
           ↓
8. LLM returns response with tool calls
           ↓
9. Execute tools (e.g., read_file, web_search)
           ↓
10. Call LLM again with tool results
           ↓
11. [Iteration loop: steps 8-10 repeat until final response]
           ↓
12. LLM returns final text response
           ↓
13. Save assistant message to memory store
           ↓
14. Publish OutboundMessage to bus
           ↓
15. Channel adapter consumes from outbound channel
           ↓
16. Adapter sends message to Telegram
           ↓
17. User receives response
```

### Tool Execution Flow

```
Agent receives tool call from LLM
           ↓
Parse tool name and arguments
           ↓
Look up tool in registry
           ↓
Check permissions (workspace restrictions)
           ↓
Execute tool
           ↓
Capture result (text, file, error)
           ↓
Filter sensitive data from result
           ↓
Format result for LLM
           ↓
Return to agent loop
```

## Key Features

### 1. Multi-Agent Support

Run multiple agents with different configurations:

```json
{
  "agents": {
    "list": [
      { "id": "assistant", "default": true },
      { "id": "coder", "model_name": "gpt-4" },
      { "id": "support", "workspace": "~/support" }
    ]
  }
}
```

Route messages using bindings based on channel, user, or context.

### 2. Sub-Agents (Spawn)

Agents can spawn sub-agents for delegation:

```
Main Agent
   ├─ spawn("research", "Find info about X")
   │     └─ Returns: "Found: ..."
   └─ spawn("analysis", "Analyze data Y")
         └─ Returns: "Analysis: ..."
```

Implementation: `pkg/tools/spawn.go`

### 3. Skills System

Reusable skill packages that extend agent capabilities:

```
~/.picoclaw/workspace/skills/
└── python-expert/
    ├── SKILL.md         # Skill definition
    └── examples/        # Example code
```

Skills are injected into system prompt when relevant.

Commands:

- `/list skills` - Show available skills
- `/use <skill>` - Force skill for next message

### 4. Cron Jobs

Schedule recurring agent tasks:

```json
{
  "cron": {
    "enabled": true,
    "jobs": [
      {
        "name": "daily-summary",
        "schedule": "0 9 * * *",
        "action": {
          "type": "send_message",
          "channel": "telegram",
          "chat_id": "123456789",
          "message": "Daily summary"
        }
      }
    ]
  }
}
```

Implementation: `pkg/cron/`

### 5. Heartbeat System

Periodic agent health checks and proactive tasks:

```
Every 30 minutes:
  - Check HEARTBEAT.md for tasks
  - Execute tasks if conditions met
  - Update state
```

Example use: Monitor system, send reminders, periodic cleanup.

### 6. Voice Integration

Transcribe voice messages using Whisper:

```json
{
  "voice": {
    "enabled": true,
    "whisper_api_url": "https://api.openai.com/v1/audio/transcriptions",
    "whisper_api_key": "${OPENAI_API_KEY}"
  }
}
```

Flow:

```
User sends voice message
    ↓
Channel downloads audio
    ↓
Transcribe via Whisper API
    ↓
Agent processes as text message
```

### 7. Vision Pipeline

Send images/files to multimodal LLMs:

```
User sends image
    ↓
Channel saves to media store
    ↓
Returns media:// URI
    ↓
Agent downloads media
    ↓
Encode as base64
    ↓
Attach to LLM request
    ↓
LLM analyzes image
```

Supports: GPT-4 Vision, Claude 3, Gemini, etc.

### 8. Steering System

Pre-seed conversation with messages before user input:

```go
type InitialSteeringMessages []providers.Message
```

Use cases:

- Few-shot examples
- Context injection
- Forced behavior

### 9. Thinking Mode

Extended reasoning for complex problems (Anthropic models):

```json
{

    "defaults": {
      "thinking_level": "medium" // none, low, medium, high

  }
```

```json
  "hooks": {
    "enabled": true,
    "processes": {
      "log-all-messages": {
        "command": "./hooks/logger.sh",
        "events": ["before_turn", "after_turn"]
      }
    }
  }
}
```

Events:

- `before_turn`: Before agent processes request
- `after_turn`: After agent responds
- `before_tool`: Before tool execution
- `after_tool`: After tool execution

## Extending PicoClaw

### Adding a New Tool

1. Create tool implementation in `pkg/tools/`:

```go
type MyTool struct {
    workspace string
}

func (t *MyTool) Name() string {
    return "my_tool"
}

func (t *MyTool) Description() string {
    return "Description of what my tool does"
}

func (t *MyTool) Parameters() map[string]any {
    return map[string]any{
        "type": "object",
        "properties": map[string]any{
            "param1": map[string]any{
                "type":        "string",
                "description": "Parameter description",
            },
        },
        "required": []string{"param1"},
    }
}

func (t *MyTool) Execute(ctx context.Context, args map[string]any) (ToolResult, error) {
    param1 := args["param1"].(string)
    // ... tool logic ...
    return TextResult{Content: "Result"}, nil
}
```

2. Register tool in agent instance (`pkg/agent/instance.go`):

```go
if cfg.Tools.IsToolEnabled("my_tool") {
    toolsRegistry.Register(tools.NewMyTool(workspace))
}
```

3. Add config option in `pkg/config/config.go`:

```go
type ToolsConfig struct {
    MyTool struct {
        Enabled bool `json:"enabled"`
    } `json:"my_tool"`
}
```

## Learning Resources

### Key Files to Study

1. **Start here**: `cmd/picoclaw/main.go` - Entry point
2. **Gateway**: `pkg/gateway/gateway.go` - System initialization
3. **Agent Loop**: `pkg/agent/loop.go` - Core processing logic
4. **Provider Interface**: `pkg/providers/types.go` - LLM abstraction
5. **Tool System**: `pkg/tools/registry.go` - Tool execution
6. **Message Bus**: `pkg/bus/bus.go` - Async messaging
7. **Channel Manager**: `pkg/channels/manager.go` - Multi-platform handling

### Architecture Patterns Used

- **Event-driven**: Message bus decouples components
- **Plugin-based**: Channels, providers, tools are pluggable
- **Factory pattern**: Dynamic provider/channel creation
- **Interface-based**: Clean abstractions (Provider, Channel, Tool, Store)
- **Pub-sub**: Message bus for async communication
- **Worker pools**: Per-channel queues with rate limiting
- **Context propagation**: Cancellation and timeouts via context.Context
- **Graceful degradation**: Fallback chains, optional capabilities

### Best Practices Observed

1. **Minimal dependencies**: Carefully chosen external libraries
2. **Interface segregation**: Optional capabilities via small interfaces
3. **Error classification**: Structured error handling for intelligent failover
4. **Configuration validation**: Early validation with helpful errors
5. **Concurrent safety**: sync.Map, atomic operations, mutexes used appropriately
6. **Resource cleanup**: Proper Close() methods and defer statements
7. **Logging**: Structured logging with context fields
8. **Testing**: Comprehensive unit tests for critical paths
